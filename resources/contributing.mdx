## Getting Started

### Prerequisites

Before contributing, ensure you have:

- **Node.js** v18 or higher
- **Foundry** for smart contract testing
- **Git** for version control
- A **Web3 wallet** for testing (MetaMask recommended)
- Basic knowledge of:
  - TypeScript and React
  - Solidity and smart contracts
  - Web3 development (Wagmi, Viem)

### Setting Up Your Development Environment

<Steps>
  <Step title="Fork the repository">
    Visit the VoteLab GitHub repository and click "Fork":
    ```
    https://github.com/Pragyat-Nikunj/Voting-Dapp
    ```
  </Step>

  <Step title="Clone your fork">
    ```bash
    git clone https://github.com/YOUR_USERNAME/Voting-Dapp.git
    cd Voting-Dapp
    ```
  </Step>

  <Step title="Install dependencies">
    ```bash
    cd frontend
    npm install
    ```
  </Step>

  <Step title="Run the development server">
    ```bash
    npm run dev
    ```
    Open [http://localhost:3000](http://localhost:3000) to see the app.
  </Step>

  <Step title="Run tests to ensure everything works">
    **Frontend tests:**
    ```bash
    npm test
    ```

    **Smart contract tests:**
    ```bash
    cd ../smart-contract
    forge test
    ```
  </Step>
</Steps>

## Development Workflow

### Creating a Branch

Always create a new branch for your work:

```bash
git checkout -b feature/your-feature-name
# or
git checkout -b fix/your-bug-fix
```

Use descriptive branch names:
- `feature/multi-option-voting`
- `fix/wallet-connection-error`
- `docs/update-api-reference`
- `test/add-admin-panel-tests`

### Making Changes

<AccordionGroup>
  <Accordion title="Frontend Development">
    **Key files and directories:**
    - `src/app/`: Next.js 16 App Router pages
    - `src/components/`: React components
    - `src/hooks/`: Custom hooks (e.g., `useMemberVote.ts`)
    - `src/constants/`: Contract ABI and address
    - `test/components/`: Vitest unit tests

    **Best practices:**
    - Use TypeScript for all new code
    - Follow existing code style and conventions
    - Ensure components are properly typed
    - Use the `useMemberVote` hook for contract interactions
    - Add tests for new components
  </Accordion>

  <Accordion title="Smart Contract Development">
    **Key files:**
    - Contract repository: [VoteLab Contract](https://github.com/Pragyat-Nikunj/Smart-Voting-Contract)
    - `src/MemberVote.sol`: Main voting contract
    - `test/`: Foundry tests

    **Best practices:**
    - Follow Solidity style guide
    - Add NatSpec comments
    - Ensure all state transitions are tested
    - Test all error conditions
    - Verify gas optimization
    - Run security checks
  </Accordion>

  <Accordion title="Testing Guidelines">
    **Frontend (Vitest + React Testing Library):**
    ```bash
    npm test
    ```

    Key testing areas:
    - Mock `useMemberVote` hook for different states
    - Test AdminPanel visibility (owner vs non-owner)
    - Verify voting button interactions
    - Ensure correct workflow state handling
    - Test transaction flows

    **Smart Contract (Foundry):**
    ```bash
    forge test -vvv
    ```

    Key testing areas:
    - All error reverts (e.g., `MemberVote__AlreadyVoted`)
    - State persistence during `Resetting` phase
    - Access control (only owner functions)
    - Vote counting accuracy
    - Entry fee validation
    - Withdrawal mechanism
  </Accordion>

  <Accordion title="Documentation">
    **What to document:**
    - New features and how to use them
    - API changes or new functions
    - Configuration options
    - Troubleshooting steps
    - Code comments for complex logic

    **Documentation locations:**
    - README.md for project overview
    - Code comments for inline documentation
    - Separate docs for user guides
    - JSDoc/NatSpec for functions
  </Accordion>
</AccordionGroup>

### Code Quality Standards

<Warning>
  All pull requests must pass CI/CD checks before merging.
</Warning>

Before submitting your contribution:

1. **Run linters:**
   ```bash
   npm run lint
   ```

2. **Run tests:**
   ```bash
   npm test
   forge test
   ```

3. **Check types:**
   ```bash
   npm run type-check
   ```

4. **Test manually:**
   - Connect wallet and test all workflows
   - Test as owner and non-owner
   - Verify state transitions
   - Check transaction confirmations

## Submitting a Pull Request

<Steps>
  <Step title="Commit your changes">
    Write clear, descriptive commit messages:
    ```bash
    git add .
    git commit -m "feat: add multi-option voting support"
    ```

    Use conventional commit format:
    - `feat:` for new features
    - `fix:` for bug fixes
    - `docs:` for documentation
    - `test:` for tests
    - `refactor:` for code refactoring
    - `style:` for formatting changes
  </Step>

  <Step title="Push to your fork">
    ```bash
    git push origin feature/your-feature-name
    ```
  </Step>

  <Step title="Create a pull request">
    1. Go to the [VoteLab repository](https://github.com/Pragyat-Nikunj/Voting-Dapp)
    2. Click "New Pull Request"
    3. Select your fork and branch
    4. Fill out the PR template with:
       - Clear description of changes
       - Motivation and context
       - Screenshots (for UI changes)
       - Testing steps
       - Breaking changes (if any)
  </Step>

  <Step title="Address review feedback">
    - Respond to reviewer comments
    - Make requested changes
    - Push updates to your branch
    - Re-request review when ready
  </Step>

  <Step title="Merge">
    Once approved, a maintainer will merge your PR. Thank you for contributing!
  </Step>
</Steps>

## Pull Request Guidelines

### PR Checklist

Before submitting, ensure:

- [ ] Code follows project style guidelines
- [ ] All tests pass (`npm test` and `forge test`)
- [ ] New tests added for new functionality
- [ ] Documentation updated (if applicable)
- [ ] No console.log statements left in code
- [ ] TypeScript types are correct
- [ ] Commit messages follow conventional format
- [ ] PR description clearly explains the changes

## Reporting Issues

### Bug Reports

When reporting a bug, include:

1. **Clear title**: Describe the issue concisely
2. **Steps to reproduce**: Detailed steps to recreate the bug
3. **Expected behavior**: What should happen
4. **Actual behavior**: What actually happens
5. **Environment**:
   - Browser and version
   - Wallet type
   - Network (mainnet, testnet, local)
   - Contract address
6. **Screenshots/logs**: Visual evidence or console errors
7. **Error messages**: Exact revert reasons from transactions

### Feature Requests

When suggesting a feature:

1. **Use case**: Explain why this feature is needed
2. **Proposed solution**: Describe how it should work
3. **Alternatives considered**: Other approaches you've thought about
4. **Impact**: Who benefits and how
